# GRAсклад — Roadmap

> Результат масштабного аудита всех 24 страниц, 14 route-файлов, 35 таблиц БД и UX-флоу.
> Дата аудита: 04.04.2026 | Текущая версия: v2.90.0

---

## 🔴 Фаза 1 — Критические фиксы (v2.91–v2.95)

> Безопасность, стабильность, защита данных. Без этого всё остальное — на песке.

### 1.1 Безопасность API
- [ ] **SQL-инъекция в earnings.js** — `period` фильтр через строковую интерполяцию вместо параметризованного запроса
- [ ] **Убрать `password_plain` из API-ответов** — сейчас пароли в открытом виде отдаются на фронт через `/staff/users` и `/staff/employees`
- [ ] **Ошибки БД утекают наружу** — `res.status(500).json({ error: err.message })` показывает имена таблиц и колонок
- [ ] **Rate-limiting** — только `/auth/login` ограничен; добавить на analytics, earnings, movements (тяжёлые запросы)
- [ ] **Ограничить `?limit=`** — сейчас `?limit=999999` загрузит всю таблицу; ввести `Math.min(limit, 1000)`

### 1.2 Производительность
- [ ] **N+1 запрос в `/movements/my-inventory`** — цикл по каждому товару вместо одного JOIN
- [ ] **Покрывающие индексы для earnings** — aggregation-запросы сканируют всю таблицу
- [ ] **Query timeout** — нет `statement_timeout`; аналитика может вешать базу бесконечно
- [ ] **Кеш авторизации — FIFO вместо LRU** — при заполнении (200 юзеров) вылетает не тот пользователь

### 1.3 Целостность данных
- [ ] **FK на `shelf_movements_s.task_id`** — нет ссылки, могут быть сироты
- [ ] **`scans_count` рассинхронизируется** — денормализованное поле не обновляется при удалении сканов
- [ ] **GRA-баланс** — если удалить записи из `employee_earnings_s`, баланс не пересчитывается; нет триггера

## 🟡 Фаза 2 — UX сотрудников (v2.95–v3.5)

> Сотрудники работают на TSD-сканерах и телефонах. Каждая секунда экономится = деньги.

### 2.1 Сканирование
- [ ] **Единый таймаут авто-отправки** — сейчас 300/350/400мс на разных страницах, сотрудники путаются
- [ ] **Глобальный audio-сервис** — каждая страница реализует `playBeep()` заново; нужна единая мут-кнопка + haptic fallback
- [ ] **«Отмена последнего скана»** — нет нигде; ошибочный скан не откатить
- [ ] **Очередь сканов** — в AssemblyPage `queueRef` принимает только 1 скан; быстрые 2-й и 3-й теряются
- [ ] **Offline-режим** — при потере сети задача теряется; нужен локальный кеш последних сканов

### 2.2 Перемещения (MovePage)
- [ ] **Подсказка при выборе режима** — 4 кнопки (Поштучно/Целиком/Перенос/Админ) без объяснений, что когда использовать
- [ ] **Предпросмотр содержимого коробки** до начала операции
- [ ] **Карта/схема склада** при выборе куда положить — сейчас надо знать ШК назначения наизусть
- [ ] **Частичная доставка** — «Мой инвентарь» и Возвраты требуют сдать всё или ничего

### 2.3 Навигация сотрудника
- [ ] **Нижняя навигация переполняется** на экранах <320px — текст обрезается, иконки налезают
- [ ] **Кнопка выхода слишком маленькая и без подтверждения** — случайный тап = потеря задачи
- [ ] **Виджет GRA-баланса обрезается** при больших числах
- [ ] **Кнопки < 44px** — Refresh, некоторые action-кнопки меньше минимума для тач-интерфейсов

### 2.4 Задачи
- [ ] **Быстрое переключение задач** — нет паузы-A → задача-B → вернуться-к-A
- [ ] **Таймер задачи** — сотрудник не видит, сколько времени уже работает
- [ ] **Фильтр задач по типу** — на MyTasksPage нет фильтров; при 30+ задачах сложно найти нужную
- [ ] **Сохранение прогресса сборки** — AssemblyPage хранит состояние в sessionStorage; если браузер перезагрузится — всё потеряно

## 🟢 Фаза 3 — UX админки (v3.5–v4.0)

> Админ и менеджеры работают с десктопа. Нужны bulk-операции, экспорт, аналитика.

### 3.1 Массовые операции
- [ ] **Выбрать несколько → действие** — задачи (пауза/отмена/переназначение), товары (изменение цены/категории), сотрудники (корректировка баланса)
- [ ] **Bulk-импорт товаров** — CSV/Excel загрузка вместо создания по одному
- [ ] **Bulk-печать этикеток** — весь стеллаж / все коробки на паллете

### 3.2 Экспорт данных
- [ ] **CSV/Excel экспорт** — перемещения, заработок, задачи, остатки
- [ ] **PDF-отчёт по инвентаризации** — итоги задачи для бумажного документооборота
- [ ] **Экспорт активности сотрудника** — для HR/бухгалтерии

### 3.3 Группировка ошибок
- [ ] **Автогруппировка одинаковых ошибок** — сейчас 10 одинаковых сканов = 10 строк
- [ ] **Тренд ошибок** — сколько ошибок в день/неделю, динамика
- [ ] **Уровни критичности** — «Не найден ШК» ≠ «Падение React-компонента»
- [ ] **Массовое «Решено»** — выбрать всё за дату, пометить решённым

### 3.4 Улучшение страницы задач (TasksPage — 1909 строк)
- [ ] **Разбить файл** — TaskList, TaskDetail, TaskModals, TaskAnalytics
- [ ] **Шаблоны задач** — ежедневная инвентаризация одной и той же полки одной кнопкой
- [ ] **SLA-мониторинг** — подсветка если задача висит > 1 часа
- [ ] **Передача задачи другому** — пауза + переназначение сотрудника

### 3.5 Улучшение склада (WarehousePage — 4283 строки)
- [ ] **Разбить файл** — WarehouseTree, WarehouseVisual, WarehouseModals
- [ ] **Тепловая карта заполненности** — цветом показать загруженность полок/паллетов
- [ ] **Drag-and-drop** в визуальном режиме — перетаскивать коробки между полками
- [ ] **Планирование ёмкости** — подсказка «куда лучше положить» по заполненности

### 3.6 Заработок
- [ ] **Подтверждение при большой корректировке** — сейчас можно списать 100K GRA без подтверждения
- [ ] **Прогноз заработка** — «при текущей скорости сотрудник заработает X за месяц»
- [ ] **Сравнение план/факт** — плановая выработка vs фактическая
- [ ] **Аудит корректировок** — кто, когда, сколько изменил (таблица в UI)

### 3.7 Live-мониторинг
- [ ] **Легенда цветов** на таймлайне — сейчас зелёный/серый/красный/жёлтый без подписей
- [ ] **Кнопка «Сбросить зум»** — после zoom/pan нет способа вернуть обзор
- [ ] **Оповещения** — перерыв > X минут, скорость сканирования упала
- [ ] **Командный вид** — агрегат по отделу/складу, не только по сотрудникам

## 🔵 Фаза 4 — Новые модули (v4.0–v5.0)

> Расширение функциональности. Новые возможности, которых пока нет в системе.

### 4.1 Уведомления
- [ ] **Центр уведомлений** — колокольчик в шапке, список событий
- [ ] **Push-уведомления** для сотрудников — новая задача, изменение приоритета
- [ ] **Email/Telegram** для админов — критические ошибки, конец смены, отчёт

### 4.2 Отчётность
- [ ] **Генератор отчётов** — выбрать период, модуль, формат → получить PDF/Excel
- [ ] **Дашборд KPI** — ежедневные/еженедельные метрики на одном экране
- [ ] **Сравнение периодов** — эта неделя vs прошлая, этот месяц vs прошлый

### 4.3 Управление сменами
- [ ] **График смен** — кто когда работает
- [ ] **Отметка начала/конца смены** — вместо ручного трекинга
- [ ] **Лимиты перерывов** — автоматический контроль: обед ≤ 30 мин, tech ≤ 10 мин

### 4.4 Складская логистика
- [ ] **Лот/партия** — если у товара есть срок годности (БАДы!)
- [ ] **FIFO-контроль** — подсказка «отгружай старое первым»
- [ ] **Минимальный остаток** — алерт когда товар заканчивается
- [ ] **Автозаказ** — интеграция с поставщиками при low stock

### 4.5 Интеграции
- [ ] **Yandex Market** — проверка ШК (по аналогии с Ozon/WB)
- [ ] **1С/бухгалтерия** — выгрузка перемещений и заработка
- [ ] **Telegram-бот** — сотрудник смотрит свои задачи/баланс в мессенджере
- [ ] **Весы/принтеры** — прямое подключение через API

### 4.6 Аналитика 2.0
- [ ] **Прогнозы** — когда закончится товар на основе скорости движения
- [ ] **Оптимизация размещения** — AI-подсказки: «товар X часто берётся, переместите ближе»
- [ ] **Паттерны ошибок** — «сотрудник Y ошибается на товаре Z» → дообучение

## 🟣 Фаза 5 — Техдолг и архитектура (параллельно)

> Делается постепенно, в фоне, вместе с фичами.

### 5.1 Рефакторинг фронтенда
- [ ] **WarehousePage.jsx** (4283 строки) → разбить на 4-5 файлов
- [ ] **TasksPage.jsx** (1909 строк) → TaskList + TaskDetail + TaskModals
- [ ] **SettingsPage.jsx** (1822 строки) → по вкладкам
- [ ] **ProductsPage.jsx** (1471 строк) → ProductList + ProductCard + ProductModals
- [ ] **Общий ScanInput-компонент** — единые правила для всех страниц сканирования

### 5.2 Бэкенд
- [ ] **Мягкое удаление (soft delete)** — `deleted_at` вместо физического DELETE; аудит
- [ ] **Валидация входящих данных** — middleware для проверки типов, длины, формата
- [ ] **Request ID / трейсинг** — корреляционный ID для отладки проблем
- [ ] **Логирование медленных запросов** — `EXPLAIN ANALYZE` в dev-режиме
- [ ] **JWT blacklist** — таблица отозванных токенов (сейчас нельзя «выгнать» пользователя)

### 5.3 Инфраструктура
- [ ] **Бэкап БД** — автоматический pg_dump по крону
- [ ] **Мониторинг** — uptime, CPU, память, медленные запросы (Grafana/простой health-check)
- [ ] **Staging-окружение** — тестировать перед деплоем на прод
- [ ] **Миграции БД** — нормальная система (не CREATE IF NOT EXISTS), откат
